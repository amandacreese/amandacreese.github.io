Accessibility Documentation

Introduction

"Accessibility is the practice of making your websites usable by as many people as possible. We traditionally think of this as being about people with disabilities, but the practice of making sites accessible also benefits other groups such as those using mobile devices, or those with slow network connections.

You might also think of accessibility as treating everyone the same, and giving them equal opportunities, no matter what their ability or circumstances. Just as it is wrong to exclude someone from a physical building because they are in a wheelchair (modern public buildings generally have wheelchair ramps or elevators), it is also not right to exclude someone from a website because they have a visual impairment. We are all different, but we are all human, and therefore have the same human rights.

Accessibility is the right thing to do. Providing accessible sites is part of the law in some countries, which can open up some significant markets that otherwise would not be able to use your services or buy your products."
– Extract from MDN web docs

WCAG Guidelines

The main guidelines for accessibility are WCAG 2.0 that were released December 2008. In June 2018 there was an additional release to introduce support for new features. These guidelines are known as WCAG 2.1 and together with 2.0 provide the entire accessibility guideline documentation.

Accessibility Classifications

There are three classifications for achieving the WCAG 2.0 and 2.1 guidelines, these are A, AA and AAA with a breakdown below of what they entail.

Level A = some impact on design

  • This is the minimum level of accessibility to be able to achieve the goal of hitting the WCAG 2.1 criteria.

Level AA = Medium impact on design

  • This is the medium ground, some areas we are already achieving this, it also has the least impact on design, and can be applied more generally.

Level AAA = High/Extreme impact on design

  • This could be classed as extreme level of accessibility, more devoted for websites that are aimed at being accessible such as RNIB etc.
  • Would more be tailored to bespoke designs as it would need to be bespoke for each website.

PrimarySites approach to accessibility is focused primarily on the website and not the CMS.

PrimarySite’s Commitment Statement for Accessibility

All Trust websites from September 1st 2018 must strive as much as possible to reach the accessibility standards as set out by the government guidance to WCAG 2.1 level AA. A decision has also been made to ensure that all SEN websites meet this criteria from the 1st August 2020. It is estimated that there will be around 3 trust/SEN websites every month.

Initial Requirements

In every website that we build, where possible we should be striving to achieve modern web standards, of course because of our customers and our customers requirements this might not always be 100% achievable, all we can do is try.
To help us achieve as much of an accessible website as possible there are some steps and practices that we need to undertake.

  • Each of our website templates uses the <html lang="en"> to set the defined language of copy.
  • Each website template should have user-scalable=no removed from the viewport meta in the header of each build.
  • HTML should be valid and validated, we can check this using W3C HTML validator. This will help check to make sure the correct code syntax has been used and tags have been opened and closed correctly.
  • It is important that any IDs used within the template are unique, conflicts in the code can happen if the ID is declared more then once on the page.
  • HTML is written in the order that it appears within the browser.
  • Semantic HTML is used for it’s correct purpose see Semantic HTML.
  • CSS and HTML are separated where possible, outside of widgets and iFrames provided to us by schools and trusts, inline styling should be avoided.
  • Within the design a clear visual hierarchy is portrayed, which goes throughout the design regardless of the screen size the design is on.

Semantic HTML

Within HTML5 there has been the introduction of semantic roles that enable assistive technologies to be able to understand straight away what role they are within the website.

  • <article> - repeated content which can be used on their own, twitter is an example where articles are used.
  • <aside> - related to main content, but should not be necessary to understand the content.
  • <button> - A button is an element used to perform actions such as,
    • submitting a form
    • opening a dialog
    • cancelling an action
    • performing a command
      An example of application is search and translate within our websites.
  • <div> and <spans> - these tags do not have any semantic meaning and can be used as general building blocks. If it’s required a role can be assigned to it using role="", but it is much better to use the correct semantic code.
  • <footer> - is a tag used to indicate the footer of a website or a website element and can be used multiple times in website markup where appropriate.
  • <header> - is a tag used to indicate the header of the website or a website element and can be used multiple times in website markup where appropriate.
  • <main> - is a tag used to indicate the main content of page.
  • <nav> - is a tag used to identify any major group of links used for navigation, it can be used multiple times in website markup where appropriate.
  • <section> - is a tag used to identify major sections of a page (this is particularly relevant to the homepage).

Heading Tags

The heading tags are used to define headings within your website. They can be used by screenreader users to skip to content based on heading, and work in a similar way way to a book index.

  • The heading tags go from <h1> to <h6>
  • <h1> defines the most important heading, whilst <h6> defines the least important.
  • Headings should never be skipped, so going from <h2> to <h4> shouldn’t happen.
  • There should only ever be 1 <h1> per page, this is normally the page title.

Additional Tags

  • <p> - this tag defines a paragraph which is a block level element. Browsers automatically add a single line space between each p that is used.
  • <ul> - this tag defines an unordered list, this is used in conjunction with the <li> tag. Without styling this is rendered as a bulleted list. It is typically used on items that don’t have a meaningful order in the list.
  • <ol> - this tag defines an ordered list of items, this is used in conjunction with the <li> rag. Without styling this is rendered as a numbered list. It is typically used on items that have a meaningful order.
  • <li> - this tag defines an item in a list and must be contained in a parent element, either a <ul> or <ol>, depending on which one they are used in depends how they are displayed.
  • <a> - this tag defines an anchor which creates a hyperlink, this could be internal, external, email addresses or anything else that a URL can address.
  • <img> - this tag embeds an image into the page, it is followed by a src attribute and, for accessibility, it requires an appropriate alt description.
  • <video>- this tag embeds a media player which supports video playback, it is followed by a src attribute to dictate where the content is coming from. This tag can be used for audio as well, but for best practice the <audio> tag should be used.
  • <audio> - this tag embeds an audio player within the website and is used to embed audio files. It is followed by a <src> attribute to dictate where the content is coming from.

ARIA Attributes

ARIA (Accessible Rich Internet Applications) is a set of attributes that can be added to web content to make it more accessible to people with disabilities.

Accessibility User Tree

An accessibility user tree is part of your websites DOM, it is made by the browser and lists all accessible objects that is then exposed to assistive technologies. To view this tree, open the Chrome Developer Tools and under the elements tab click the accessibility tab. ARIA works by changing the standard DOM tree.

Where semantic HTML is used, an element doesn’t need any additional role or ARIA label unless the label is confusing.

More information about the accessibility tree can be found through Google Developer Docs

Aria Tags

  • aria-label - is an attribute that can be used when you have some kind of visual indication to an items purpose, or where the message might get confusing to the screen reader user.
  • aria-labelledby - allows us to create a relationship between elements regardless of their relationship within the DOM. For example a section will not necessarily have a screen reader friendly description however by using aria-labelledby and giving the unique id to an element we can give a friendlier name. By using this method duplication can be avoided. The example that follows uses this ARIA label and if you update the content of the <h2> tag you update the section tag automatically. <section aria-labelledby="meet-the-staff"> <h2 id="meet-the-staff">Meet the staff</h2>. It is important to remember that each aria-labelledby entry must be unique.
  • aria-describedby - is an attribute that can be used to provide an accessible description, for example a video or audio transcript.
  • aria-hidden - this is an attribute that can be combined with true ( aria-hidden="true") to hide something from the accessibility tree, this maybe something that has no value to the screen reader user, is duplicated content or is an of screen or collapsable element.

Images

An image can be used in multiple ways in website design and layout. It could be used to convey a message, or for a purely aesthetic reason. Images are broken down into two main categories.

Decorative

  • Images that don’t serve a purpose such as a dividing line graphic.
  • A background or for aesthetic purposes, such as our illustrated website designs.
  • Contains no important informative text.

Functional

  • Present for a reason, emotion, context to a story.
  • Icons, buttons, product images.
  • Text, infographics, data charts.

Image Section Summary

  • Every image that is embedded through the website theme requires an alt tag even if it is an empty alt tag because the image is decorative, this is because a screen reader will skip alt="", where as if it doesn’t have an alt tag at all the screenreader will make an attempt at reading the image.
  • An alt="" tag is important for portraying a text representation of an image, it must be descriptive but you do not need to include in the description that this is an image as that is implied already.
  • Screen readers will only really read up to 125 characters of an alt text tag. If the description is going to be longer, because the image is more complex, a separate p tag set with the class of sr-only following the content is more appropriate.
  • Different types of images have to be handled differently.
  • Any background image that is going to have text as an overlay needs to have a background colour associated with it, this is in case the image fails to load, the text will still be legible.

Links

Links are used to guide the user around or away from the website, they may form part of a user journey, aiming to get a user from a to b, or they may be part of the bigger website picture.
From a visual hierarchy it is important that there is a difference in style between what is a link to a part of the website or a part of the page, compared to what would be classed as a button which is interactive but doesn’t necessarily navigate the user to a different part of the website.
The importance into spending time into links and how they are presented to the user, can make a user journey for any type of user really pleasant to experience.

Link Best Practices

  • Links should not have any fixed height or fixed width attached to it, as part of accessibility certain users will enable zoom in features to be able to easily read the content of the element. Heights or widths will cause these elements to break.
  • Links should not contain <p></p> tags, this is prominently for screenreader purposes. A screenreader will encounter these and reads out what ever is within the content as a paragraph potentially confusing the user.
  • It could be considered a poor user experience if any sort of media, or any link takes the user off site, this can be fixed in the alt tag by adding in some additional code to explain in more detail what may happen to the user, the screenreader technology will read this out before the user confirms if they want to carry on further, an example of this could be,
    <a href="https://www.google.co.uk/" alt="Visit the Google website external site" rel="noopener" target="_blank">Visit the google website</a>
  • Any link should have a minimum hit area of 44px, this is more aimed at users with motor function issues and allows a user a better chance of hitting the link area. Several examples of this. can be found in the codepen below.

Short Text Links

Short text for accessibility could be confusing to the user, but it also could have a negative affect on a website’s SEO. Sometimes however this isn’t always achievable as putting a full understandable title could compromise the design and layout.

Examples of short text links are

  • Click here
  • Read more
  • Info

This is completely ok, as from a visual standpoint the link will be grouped with the relevant item it is representing and more detailed information can be placed within the alt tag.

<a href="{% calendar_grid_url %}" alt="Visit our Calendar to Read More">Read more</a>

Where possible though, we should always try and rephrase the link text so the purpose is clear.

<a href="{% calendar_grid_url %}" >View Our Calendar</a>

Image Based Links

Images are sometimes used in place of a text link, an example of this is the home link. This may not have any text associated with it as it is just using an image of the school or trust logo.

  • Where an image is used as a link, alt text must be used to describe the image and link. <a href="/" class="home-link" title="School Name" alt=”School Name Home”></a>
  • In the alt description of the link you do not need to indicate that it is a link or that it is an image as these are both implied by the screenreader.

Where an element is required that is a visual indication of an action, such as a burger menu, an aria label attribute should be included to help assistive technologies indicate its purpose.

  • <button aria-label="menu" class="burger"></button>

Navigation

As part of evolving designs and user journeys there are now several different types of navigations and navigation roles.
The different types of navigation help a variety of users navigate the website.

  • Main Navigation, helps users find the important pages within the top level navigation. This can be presented in a side bar, header or burger type navigation.
  • Sitemap, this is a different pattern of navigation that should display everything within the site, this is a much more simple format and gives a user a much more visual outlay of what is within the website.
  • Breadcrumbs, these show the user where they are within the hierarchy of the page, particularly useful for websites that have multi levels.
  • Related Navigation, this shows the user what other pages are within the selected navigation menu. This should be top level pages only and not subpages as this will dilute the message for the user.

With all types of navigation, it is important to follow some guidelines to help users of different abilities navigate the website.

  • Use the navigation Semantic HTML see semantic HTML
  • Set the correct aria-label for each navigation, this is particularly important if there is more then one instance of navigation on the website.
    • The main navigation can be set <nav class="main-nav" aria-label="main"></nav>,
    • The related navigation can be set, <nav class="related-nav" aria-label="related"></nav>
  • In a <nav></nav> setting, appropriate semantic elements should be used, most navigation patterns rely on unordered lists and anchor tags.

Keyboard Operation

As part of supporting different types of accessibility, it is highly recommended that a website can be operable by keyboard, the user types that this supports can be broken down into three categories,

  • A Power User
    This is a type of user that prefers the speed of navigation through keyboard movement.
  • A User with Movement Disorders
    This is a type of user that uses keyboard operation to help navigate with better procession.
  • Other Types of Assistive Technology
    Whilst strictly not a user, a lot of the other assistive technologies that are in use, such as voice recognition, generates keystrokes that act as keyboard usage.

To help facilitate these users the following should be recognised.

  • In each template there should be a stylised a:focus element, this will be set by default to a certain border width which shouldn’t change, what will change is the colour of the border, this should be the opposite of the colour of the design to stand out.
  • All interact-able elements should be focusable.
  • There should be no keyboard traps. A. keyboard trap is where a user can get stuck in a box and is unable to navigate their way out.
  • Have a logical tab order, this will follow source order if you don’t do anything. Which should be read left to right, top to bottom, this is why it’s important to make sure your HTML follows the semantic order.

Tabindex

The tabindex is a global attribute that can be applied to an element and indicates that that element can be focusable within the content. This is normally done on elements that may not naturally fall within the accessibility tree.

There are three main values within the tabindex attribute.

  • tabindex="-1" is a negative value, means that the element is not reachable via keyboard navigation.
  • tabindex="0" means that the element should be focusable in sequential keyboard, this means, that this element will keep its natural state within the order it has been placed in. If an element shares the same tabindex number, it will go in source order.
  • tabindex="1" this value can be between 1 and 1000 and this elevates the priority in tab order. Normally where this may need to be used, it indicates that there could possible be an issue with HTML and CSS so it would probably be easier to amend this.
  • tabindex should not be used as a replacement for using the correct syntax, e.g. a div element for a button.

Colour Contrast

Part of the WCAG 2.1 guidelines level AA requires colours for normal text to reach 4.5:1 or above and for large text to reach 3:1. The WebAIM contrast checker is a good tool for this.

Large text is defined at 24px or 2.4rem or higher.

Logos and brands are exempt from the contrast rules as well as any images of text.

Input controls, buttons, icons and charts need to have a contrast ratio of 3:1 the exception to this is if the control is disabled.

Additional Contrast Tools

Chrome currently has tools to help adjust colours within this threshold. If you select a colour that is already in use, it will display if the colour passes contrast, click on the down arrow next to the contrast ratio, it will expand and reveal the thresholds, if you click on the Aa button it will introduce two lines into the colour palette above to guide you to where you need to be for the colour to pass contrast checker.

In most instances, correct colours will be given to you by the designer, however if adding something new, or in process of retrofit, this tool is particularly useful.

Video

Videos on websites are becoming more and more common place as a way of portraying in a more impactful way something that is important. This could be a welcome from the headteacher video, a school tour, or a homepage video background. Each of these types needs to be understood carefully to understand the best ways of allowing all users a good way of experiencing this video.

  • Audio description of visual information - This is used to provide information to a video that is visual only, this maybe a school tour or a homepage video background. It is important in these situations to describe the content for all users but only if the video has an important message to portray.
  • Captions - Also referred to as subtitles, provide a story for users who are deaf or hard of hearing. They also provide an opportunity for people who may not be able to have their sound on to understand what is going on in the video. They are synchronised with the video audio and are normally displayed when the user turns them on.
  • Transcripts - A transcript is a body of text used to describe the video, it’s not in addition to the video, as most people who use this technology will not use the video aspect. It is supposed to be a basic text version of speech and surrounding interactions that are important in helping the user understand the content.
  • Sign Language - Where hand and arm movements along with facial expressions and body positions are used to convey meaning. This is a difficult thing to do and can be quite costly to maintain, most accessibility policies do not require this as the other ways should be sufficient enough to convey meaning.

It is important to stress, that PrimarySites responsibility only applies to a) providing captions for Videos we have produced and b) Transcripts we implement that are provided by the school or trust (for example Videos they have shot and PrimarySite upload). We do not need to worry about video’s uploaded by a school or Trust through the content block. We can provide this service on request and this will become a chargeable job.

Video Requirements

With each video uploaded through the theme we need to make sure that

  • Each video has a descriptive aria-label.
  • Each video has a caption or transcript where appropriate.
  • Any video does not flash more then 3 times a second.

Captions

Captions need to be provided on videos that are heavy on the audio content, this particularly applies to interview type videos, and videos with a lot of spoken words.

  • Captions need to be written in WEBVTT format this format dictates the text between a certain period of time as set by the interpreter.
  • Each queue block, which is a period of time, must contain no more than 32 characters to allow it to be rendered correctly.
  • For ease of implementation videos can be uploaded through YouTube privately to create the WEBVTT format.

Descriptive Transcripts

A transcript needs to be provided for videos that are visually important or set a context on the website. These could be a school tour, or a video background website.
This transcript should be provided by PrimarySite if it’s a video we have created, or by the school or trust if it’s a video that they have created.

  • An example of how this that could be implemented is <video controls src="" aria-label="around the school" aria-describedby="transcript"></video><div id="transcript" class="sr-only">Description goes here</video> with the class sr-only allowing the content to be hidden from the desktop user but not from the screen reader user.

Updates

WCAG 2.0 was released 11th December 2008, WCAG 2.1 was released June 5th 2018 as an addition to 2.0. This means that the guidelines are nearly 12 years old. The new version is currently being written, and is under the project codename silver.

First draft of new guidelines is going to be released in late 2021 and final guidelines are to be released late 2022.

This release will also move away from the A, AA and AAA classification and changing to bronze, silver and gold classification, which will be a points based system instead. This gives websites the opportunity to work towards accessibility for the majority of people instead of a pass fail exercise.

More information about this can be found on My Accessible Website
